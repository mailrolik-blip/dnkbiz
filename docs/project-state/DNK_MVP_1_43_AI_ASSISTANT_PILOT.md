# DNK MVP 1.43: AI Assistant Pilot Request Flow

## Зачем добавлен AI-помощник

MVP 1.43 добавляет первый рабочий вход в пилот AI-помощника для малого бизнеса внутри DNK. Это не отдельный промо-лендинг и не полноценный AI-раздел, а короткий app-like flow: пользователь из ЛК описывает задачу, выбирает набор функций будущего помощника, отправляет заявку и видит демо-кабинет с понятным сценарием работы.

Цель пилота - собрать реальные боли собственников и подготовить ручную очередь заявок для настройки сценариев в n8n.

## Пользовательский flow

1. Пользователь открывает `/lk`.
2. В ЛК видит карточку "AI-помощник" как рабочий инструмент.
3. Если заявки нет, CTA ведёт на короткий разбор.
4. Если заявка уже есть, ЛК показывает статус последней заявки и даёт действия "Открыть демо-кабинет" и "Отправить новую задачу".
5. На `/ai-assistant` пользователь проходит wizard:
   - тип бизнеса;
   - главная боль;
   - функции помощника;
   - каналы общения;
   - контакт.
6. Шаг функций помощника является multi-select. Пользователь собирает набор возможностей, а не выбирает одну функцию.
7. Для авторизованного пользователя `userId`, имя и email берутся из текущей сессии. Контакт для этой заявки можно уточнить отдельным полем.
8. Для гостя имя и контакт запрашиваются вручную.
9. После отправки пользователь попадает в рабочую область AI-помощника.
10. По умолчанию открыта вкладка "Чат"; заявка, функции, база знаний и статус доступны во вторичных вкладках.

## Что входит в пилот

- Страница `/ai-assistant` с интерактивным wizard.
- Multi-select функций помощника:
  - первичные ответы клиентам;
  - сбор заявки;
  - запись / напоминания;
  - база знаний;
  - подготовка инструкций;
  - разбор проблемы;
  - контроль повторяющихся задач;
  - помощь администратору;
  - помощь в обучении сотрудников;
  - уведомления собственнику;
  - передача заявки в таблицу/CRM.
- Сохранение заявки в `AiAssistantRequest`.
- Привязка заявки к `userId`, если пользователь авторизован.
- Рабочая область помощника после отправки заявки.
- Чат-вкладка с полем ввода и состоянием "помощник отвечает".
- MVP bridge к n8n chat webhook без хранения истории чата в DNK.
- Карточка AI-помощника в ЛК.
- Блок "AI-заявки" в админке.
- Подготовка payload и краткого ТЗ для n8n.

## Admin flow

В `/admin` добавлен блок "AI-заявки" / "Очередь заявок на помощника".

Администратор работает с master-detail layout:

- слева компактная очередь заявок;
- справа детали выбранной заявки.

В списке заявок видно:

- id заявки;
- имя / email или контакт;
- тип бизнеса;
- главную боль;
- статус;
- дату;
- компактный n8n status;
- максимум две функции и `+N`.

В деталях заявки видно:

- имя и контакт для связи;
- пользователя DNK / email, если заявка привязана к аккаунту;
- тип бизнеса;
- главную боль;
- выбранные функции помощника отдельными бейджами;
- каналы;
- статус;
- `payload`;
- `n8nStatus` и `n8nResponse`.

`payload` и `n8nResponse` скрыты по умолчанию в collapsed details.

Доступные действия:

- сменить статус: `NEW`, `REVIEWED`, `IN_PROGRESS`, `READY`, `CLOSED`;
- открыть детали заявки;
- скопировать краткое ТЗ для n8n;
- открыть n8n workflow по ссылке, если задан `N8N_ASSISTANT_WORKFLOW_URL` или `N8N_WORKFLOW_URL`.

## Хранение

Добавлена минимальная Prisma-модель `AiAssistantRequest`.

Она хранит только очередь заявок пилота:

- `userId` optional;
- имя и контакт;
- тип бизнеса;
- главная боль;
- задачи и каналы в JSON;
- комментарий;
- статус;
- `n8nStatus`;
- `n8nResponse`;
- исходный `payload`;
- даты создания и обновления.

Это не CRM, не AI-память и не база знаний бизнеса.

## n8n

Env:

```env
N8N_ASSISTANT_WEBHOOK_URL=""
N8N_ASSISTANT_CHAT_WEBHOOK_URL=""
N8N_ASSISTANT_WORKFLOW_URL=""
```

Админка также поддерживает fallback-ссылку:

```env
N8N_WORKFLOW_URL=""
```

Поведение отправки:

- заявка всегда сначала сохраняется в БД;
- если `N8N_ASSISTANT_WEBHOOK_URL` не задан, заявка остаётся видимой в админке с webhook status `NOT_CONFIGURED`;
- если webhook задан, DNK отправляет payload в n8n;
- ответ или ошибка n8n сохраняются в `n8nResponse`;
- `n8nStatus` показывает `SENT`, `FAILED`, `FAILED_<status>` или `NOT_CONFIGURED`.

Поведение chat webhook:

- если `N8N_ASSISTANT_CHAT_WEBHOOK_URL` задан, сообщение пользователя отправляется в n8n;
- payload содержит `userId`, `requestId`, user name/email, `message`, выбранные функции, тип бизнеса, главную боль, каналы и `sessionId`;
- ответ n8n показывается в чате;
- если chat webhook не задан или не отвечает, DNK показывает безопасный mock-ответ: "Пилотный помощник ещё не подключён. Мы уже получили вашу задачу и подготовим тестовый сценарий.";
- история чата не сохраняется в DNK в рамках MVP 1.43.

## Что не входит в MVP

- Настоящий AI-чат внутри DNK.
- Встроенная AI-память или база знаний в Prisma.
- n8n iframe.
- Полноценная CRM или pipeline продаж.
- Автоматическая настройка n8n из DNK.
- Изменения T-Bank/payment/access logic.
- Изменения курсов, уроков и seed.

## Что считать успешным тестом

- Пользователь проходит wizard и выбирает несколько функций помощника.
- Авторизованная заявка сохраняется с `userId`.
- Guest-flow продолжает работать без авторизации.
- После отправки пользователь видит рабочую область помощника, где по умолчанию открыт чат.
- Summary во вкладке заявки выводится без склеенных подписей.
- Заявка сохраняется даже без webhook.
- Администратор видит заявку в `/admin`.
- Администратор работает с очередью заявок и detail panel.
- Администратор может сменить статус и скопировать ТЗ для n8n.
- При настроенном webhook payload уходит в n8n, а результат фиксируется в заявке.

## Future

- self-host/OEM/embedded editor для n8n можно рассмотреть отдельно, если появится безопасный и поддерживаемый способ встроить редактор workflow.
- В MVP 1.43 embedded editor и iframe n8n не реализуются: DNK хранит заявку, ТЗ, статус и историю подготовки, а рабочее пространство с нодами открывается во внешней вкладке n8n.
- ЛК остаётся рабочим: курсы, оплаты, доступы и прогресс не меняются.
